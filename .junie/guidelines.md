### 1. Архитектурные правила
- **Простота**: Не вводить лишние абстракции до второй реальной реализации. Никаких plugin systems или interfaces everywhere в MVP.
- **Слои**:
    - `cli` — разбор аргументов, запуск use-case, вывод.
    - `config` — загрузка и проверка конфига.
    - `validate` — доменная логика проверок.
    - `pack` — staging/copy/archive.
    - `report` — вывод в text/json.
- **Чистота**: Доменные пакеты не должны использовать `fmt.Println`. Только CLI-слой отвечает за вывод пользователю.
- **Зависимости**: Использовать стандартную библиотеку, если её достаточно. Минимум внешних зависимостей.
- **API**: Экспортировать только необходимое. Избегать "god packages".

### 2. Код и Стиль (Go)
- **Идиоматичность**: Маленькие пакеты и функции, явная обработка ошибок, отсутствие магии, читаемые имена.
- **Контекстные ошибки**: Всегда оборачивать ошибки с контекстом (напр., `fmt.Errorf("load config %q: %w", path, err)`).
- **Никаких panic**: Использовать `error` для всех рабочих сценариев. `panic` только для действительно невозможных ситуаций.
- **Side effects**: Явно документировать файловые операции. Функции должны соответствовать именам.
- **Читаемость**: Предпочитать понятный код компактному. Заменять сложные regex простыми проверками, где возможно.

### 3. Тестирование
- **Поведение**: Тестировать контракт и поведение, а не детали реализации.
- **Изоляция**: Использовать `t.TempDir()` для файловой логики. Никаких внешних путей.
- **Интеграционные тесты**: Обязательны для happy path и основных ошибок (valid config, missing sourceDir, exclusions).
- **Скорость**: Тесты должны быть быстрыми, без сети, внешних сервисов и случайности.

### 4. UX и CLI
- **Понятные ошибки**: Ошибки должны указывать на конкретную проблему (напр., `config error: module.id is required`) и помогать её исправить.
- **Summary**: После выполнения выводить краткий итог (кол-во ошибок/предупреждений, что делать дальше).
- **Лаконичность**: Не заливать логами. Подробности только при необходимости.

### 5. Конфигурация и Валидация
- **Структура**: Группировать поля в YAML по секциям, не разбрасывать в корне.
- **Дефолты**:
    - `outputDir` -> `./dist`
    - `stagingDir` -> `./.bxpack/staging`
    - `archiveName` -> `"{module.id}-{module.version}.zip"`
- **Валидация**: Каждое правило — отдельный тестируемый валидатор. Результаты должны иметь коды (напр., `MODULE_ID_INVALID`).
- **Severity**: Четкое разделение на `error` (блокирует build), `warning` и `info`.

### 6. Сборка (Packaging)
- **Порядок**: Сначала полная валидация, затем сборка.
- **Staging**: Всегда использовать промежуточную директорию (staging) перед архивацией.
- **Исключения**: Использовать единый механизм исключений для валидации и копирования.
- **Рекурсия**: Не включать `outputDir` и сам создаваемый архив в состав сборки.

### 7. Документация и Совместимость
- **Локализация**:
    - Все тексты, видимые пользователю (README, help, описания команд, сообщения об ошибках и предупреждениях), должны быть на русском.
    - Примеры команд и конфигурации должны сопровождаться пояснениями на русском.
    - Шаблонный `.bxpack.yml`, создаваемый командой `init`, должен содержать комментарии на русском.
- **Технические идентификаторы**:
    - Код, имена пакетов, структур, поля JSON и YAML оставить на английском.
    - Коды ошибок/валидаторов оставить на английском в верхнем регистре (напр., `MODULE_ID_INVALID`).
- **Стиль**: Коротко, понятно, без канцелярита.
- **Примеры**: Любая новая фича должна иметь пример команды и вывода.
- **Актуальность**: README должен отражать реальное поведение.
- **Совместимость**: Не ломать формат конфига и JSON-вывод без веских причин. При изменениях обновлять README и добавлять migration notes.
