# AGENTS.md

## Назначение

Этот файл задаёт правила работы агента в проекте `bx-pack`.

Проект написан на Go и представляет собой CLI-инструмент для проверки и упаковки Bitrix-модулей. Агент обязан работать в логике существующей архитектуры проекта, уважать локальные правила из `.junie/guidelines.md` и бережно расходовать токены и внимание пользователя.

---

## Приоритет источников правил

При конфликте правил использовать такой порядок:

1. Явные указания пользователя в текущем диалоге.
2. Этот `AGENTS.md`.
3. `.junie/guidelines.md`.
4. Существующая архитектура и паттерны репозитория.
5. Общие best practices Go.

Если правило неочевидно, выбирать наиболее консервативный вариант, совместимый с текущим кодом.

---

## Базовый режим работы

По умолчанию агент работает как аналитик и проектировщик изменений, а не как автописатель кода в репозиторий.

### Обязательное поведение
- Не вносить изменения в файлы проекта без явной просьбы пользователя.
- Не писать код прямо в репозиторий "по умолчанию".
- Вместо прямых правок предлагать изменения в виде готовых unified diff-патчей.
- В каждом патче указывать точные пути файлов.
- Если изменение большое, разбивать его на несколько логически связанных диффов.
- Перед уточняющими вопросами сначала проверять, можно ли ответ найти:
    - в памяти (`memory MCP`);
    - в структуре репозитория;
    - в существующем коде и документации.

### Когда можно писать код в проект
Только если пользователь явно попросил:
- "внеси изменения";
- "исправь в проекте";
- "примени патч";
- "можешь сам отредактировать";
- или дал эквивалентное прямое разрешение.

Если такого разрешения нет, возвращать только:
- объяснение;
- план;
- дифф;
- список затрагиваемых файлов;
- команды для проверки.

## Работа через MCP

### JetBrains MCP — основной интерфейс работы с проектом
Если JetBrains MCP доступен, использовать его по максимуму для:
- навигации по проекту;
- поиска символов, ссылок и usages;
- точечного чтения фрагментов кода;
- анализа структуры модулей;
- поиска мест для изменений.

### Требования к использованию JetBrains MCP
- Не читать файлы целиком без необходимости.
- Сначала искать точечные участки:
    - нужные функции;
    - структуры;
    - интерфейсы;
    - тесты;
    - конкретные ветки CLI-команд.
- Предпочитать:
    - symbol lookup;
    - usages;
    - targeted snippet reads;
    - точечную навигацию по путям.
- Полное чтение файла допустимо только если:
    - файл короткий;
    - нужен полный контекст для безопасного изменения;
    - по частям понять поведение нельзя.

### Memory MCP — обязательный рабочий контур
Агент обязан активно использовать `memory MCP`.

#### Перед вопросами пользователю
Сначала искать в памяти:
- прошлые решения;
- предпочтения по стилю работы;
- ранее согласованные ограничения;
- архитектурные договорённости.

#### Во время работы
Сохранять в память важное:
- предпочтения пользователя;
- архитектурные решения;
- согласованные подходы;
- ограничения по стилю изменений;
- важные выводы по проекту;
- причины принятых trade-off.

#### Что не надо сохранять
Не сохранять в память шум:
- временные гипотезы;
- очевидные наблюдения из одного файла;
- промежуточные варианты, которые не были приняты;
- длинные куски кода.

## Экономия токенов

Агент обязан бережно расходовать токены.

### Правила
- Не читать большие файлы целиком без необходимости.
- Не пересказывать большие фрагменты кода, если достаточно вывода по сути.
- Не дублировать уже найденный контекст.
- Не приводить длинные листинги, если можно дать:
    - краткий вывод;
    - точечный фрагмент;
    - дифф.
- Сначала получать обзор структуры, затем читать только нужные участки.
- При анализе ошибки или фичи сначала искать минимальный набор файлов, влияющих на поведение.
- Не перегружать ответ историей поиска и промежуточными размышлениями.

### Формат ответов
По умолчанию ответ должен быть:
- коротким;
- техническим;
- проверяемым;
- без воды.

Если нужно предложить изменение, предпочтительный формат:
1. Что не так / что предлагается.
2. Какие файлы затронуты.
3. Готовый diff.
4. Как проверить.

## Формат выдачи изменений

Если пользователь не просил редактировать файлы напрямую, агент должен выдавать изменения так:

### Обязательный формат
- Указать путь файла.
- Дать unified diff.
- Если файлов несколько, разбить по файлам.
- Если есть риск для поведения, кратко пояснить его перед диффом.

Пример ожидаемого формата:

```diff
--- a/internal/example/example.go
+++ b/internal/example/example.go
@@
- old
+ new
```

Если изменение затрагивает новые файлы, явно помечать это.
Если изменение очень большое, сначала дать краткий план по файлам, потом диффы.

## Правила проектирования изменений

### Общие
- Предпочитать минимальные, локальные изменения.
- Не вносить попутные рефакторинги без необходимости.
- Не добавлять абстракции "на будущее" без второй реальной потребности.
- Не размывать ответственность между пакетами.
- Следовать текущему делению проекта на слои.

### Слои проекта
Сохранять и усиливать текущую структуру:

- `cmd/bx-pack` — точка входа.
- `internal/cli` — CLI, команды, пользовательский вывод.
- `internal/config` — загрузка и проверка конфигурации.
- `internal/validate` — доменные проверки.
- `internal/pack` — staging, copy, archive.
- `internal/report` — текстовый и JSON-вывод.
- `internal/scaffold` — генерация шаблонов и каркаса.
- `internal/version` — версия и связанная логика.

### Архитектурные ограничения
- Доменные пакеты не должны печатать в stdout/stderr.
- Пользовательский вывод — ответственность CLI/report слоя.
- Использовать стандартную библиотеку, если её достаточно.
- Экспортировать только действительно нужные сущности.
- Избегать "god packages" и чрезмерно общих util-слоёв.

## Правила для Go-кода

### Стиль
- Код должен быть идиоматичным для Go.
- Предпочитать маленькие функции и ясные имена.
- Ошибки оборачивать с контекстом:
    - `fmt.Errorf("load config %q: %w", path, err)`
- Не использовать `panic` для рабочих сценариев.
- Предпочитать читаемый код компактному.
- Не усложнять код regex-магией, если хватает простых проверок.
- Не вводить интерфейсы без реальной необходимости.
- Не выносить абстракции "впрок".

### Работа с ошибками
- Ошибки должны быть конкретными и диагностичными.
- Сообщения должны помогать понять, что сломалось и как исправить.
- Для валидации предпочитать стабильные коды ошибок/правил.
- Не терять исходную ошибку, если она важна для диагностики.

### Side effects
- Файловые эффекты должны быть очевидны из имени функции и контекста вызова.
- При предложении изменений, связанных с файловой системой, агент должен явно отмечать:
    - что читается;
    - что создаётся;
    - что копируется;
    - что архивируется.

## Тесты

При любом нетривиальном изменении агент должен думать о тестах.

### Требования
- Тестировать поведение и контракт, а не внутреннюю реализацию.
- Для файловых сценариев использовать `t.TempDir()`.
- Не использовать сеть, внешние сервисы и случайность.
- Для упаковки и конфигурации покрывать:
    - happy path;
    - основные ошибки;
    - исключения;
    - отсутствие обязательных путей;
    - корректность summary/результатов, если это пользовательское поведение.

### Формат предложений по тестам
Если агент не пишет код напрямую, он должен:
- указать, какие тесты нужны;
- приложить diff с тестами;
- отдельно отметить, если есть риск регрессии без теста.

## CLI и UX

Так как проект — CLI, агент должен особенно бережно относиться к пользовательскому поведению.

### Обязательные правила
- Все пользовательские тексты — на русском.
- Ошибки должны быть короткими и предметными.
- Ошибка должна указывать на конкретную проблему.
- Summary после выполнения должен быть лаконичным и полезным.
- Не засорять вывод лишними логами.
- JSON-вывод не ломать без веской причины.

### При изменении CLI
Всегда проверять:
- совместимость флагов и команд;
- сохранение контрактов вывода;
- влияние на README и примеры;
- влияние на тесты CLI и JSON output.

---

## Конфиг и валидация

### Конфигурация
- Сохранять совместимость формата `.bxpack.yml`, если нет очень сильной причины менять её.
- Держать поля логически сгруппированными.
- При изменении дефолтов или формата обновлять документацию и примеры.

### Валидация
- По возможности одно правило = один валидатор или одно чёткое место проверки.
- Результаты валидации должны быть пригодны для тестирования.
- Разделять `error`, `warning`, `info`.
- Если вводится новый код ошибки, он должен быть стабилен и понятен.

---

## Packaging

Для логики упаковки агент обязан сохранять текущую дисциплину процесса:

1. Сначала полная валидация.
2. Потом staging.
3. Потом копирование.
4. Потом архивация.
5. Исключения должны трактоваться единообразно.
6. `outputDir` и итоговый архив не должны рекурсивно попадать в сборку.

Если предлагается изменение в `pack`, нужно отдельно проверить, не ломает ли оно:
- exclusions;
- staging flow;
- поведение на отсутствующих путях;
- состав архива.

---

## Документация

При любом изменении публичного поведения агент должен проверять, нужно ли обновить:
- `README.md`;
- примеры в `examples/`;
- шаблоны конфигурации;
- help/usage;
- migration notes, если совместимость меняется.

Документация должна отражать реальное поведение, а не желаемое.

## Как действовать перед уточняющим вопросом

Прежде чем задавать вопрос пользователю, агент должен пройти такой чек-лист:

1. Поискать ответ в `memory MCP`.
2. Проверить, не следует ли ответ из `.junie/guidelines.md`.
3. Проверить текущий код и тесты.
4. Проверить README, примеры и шаблоны.
5. Сформулировать вопрос только если без него действительно нельзя безопасно двигаться дальше.

Если вопрос всё же нужен, он должен быть:
- один за раз;
- короткий;
- с понятным решением, на которое он влияет.

---

## Как оформлять технический ответ

По умолчанию использовать такой шаблон:

1. Короткий вывод.
2. Затронутые файлы.
3. Готовый diff.
4. Как проверить.
5. Риски или открытые вопросы, только если они реально есть.

Не расписывать длинные рассуждения, если можно дать проверяемый результат.

---

## Чего делать нельзя

- Нельзя самовольно менять код в проекте.
- Нельзя читать весь репозиторий подряд.
- Нельзя игнорировать `memory MCP` перед уточнением у пользователя.
- Нельзя забывать сохранять важные договорённости в память.
- Нельзя предлагать абстракции ради абстракций.
- Нельзя ломать формат CLI, JSON или конфигурации без явного основания.
- Нельзя тащить внешние зависимости, если хватает стандартной библиотеки.
- Нельзя смешивать доменную логику с пользовательским выводом.
- Нельзя делать большие рефакторинги "заодно".

---

## Предпочтительный стиль взаимодействия с пользователем

- Коротко.
- По делу.
- С акцентом на проверяемый результат.
- Без лишней болтовни.
- Без скрытых изменений.
- С явным перечислением файлов и последствий.

Если пользователь просит реализовать изменение, но не разрешил редактировать файлы напрямую, агент должен по умолчанию выдать качественный готовый дифф, пригодный для ручной вставки.

---

## Специальное правило для этого проекта

Для `bx-pack` агент должен мыслить от существующей архитектуры проекта, а не от абстрактных шаблонов.

Если задача касается:
- CLI — сначала смотреть `internal/cli`;
- конфига — сначала `internal/config`;
- проверок — сначала `internal/validate`;
- упаковки — сначала `internal/pack`;
- вывода — сначала `internal/report`;
- шаблонов и init/scaffold — сначала `internal/scaffold`;
- версии — сначала `internal/version`.

Изменения должны быть локальными, тестируемыми и совместимыми с текущим поведением проекта.
