# CLAUDE.md

## Проект

- **Название:** bx-pack
- **Язык:** Go
- **Тип:** CLI-инструмент для проверки и упаковки Bitrix-модулей
- **Архитектура:** Слоистая структура с элементами DDD

---

## Архитектурные слои

Сохранять и усиливать текущую структуру:

- `cmd/bx-pack` — точка входа
- `internal/cli` — CLI, команды, пользовательский вывод
- `internal/config` — загрузка и проверка конфигурации
- `internal/validate` — доменные проверки
- `internal/pack` — staging, copy, archive
- `internal/report` — текстовый и JSON-вывод
- `internal/scaffold` — генерация шаблонов и каркаса
- `internal/version` — версия и связанная логика

### Ограничения
- Доменные пакеты не должны печатать в stdout/stderr
- Пользовательский вывод — ответственность CLI/report слоя
- Использовать стандартную библиотеку, если её достаточно
- Экспортировать только действительно нужные сущности
- Избегать "god packages" и чрезмерно общих util-слоёв

## Стиль и конвенции

### Go-way нейминг
- camelCase для функций/переменных
- PascalCase для экспортируемых сущностей
- snake_case для файлов
- Строгая типизация везде (по умолчанию в Go)

### Комментарии
- Для неочевидного кода
- Для экспортируемых методов (godoc)
- Для пакетов (package documentation)

### Ошибки
- Оборачивать с контекстом: `fmt.Errorf("load config %q: %w", path, err)`
- Конкретные и диагностичные сообщения
- Не использовать `panic` для рабочих сценариев
- Не терять исходную ошибку, если она важна для диагностики

### Side effects
- Файловые эффекты должны быть очевидны из имени функции
- При изменениях явно отмечать: что читается, создаётся, копируется, архивируется

## Команды

Все команды идут через Taskfile (`Taskfile.yml`), а не напрямую через `go`/`golangci-lint`:

```bash
task test    # go test -race ./...
task lint    # golangci-lint run ./...
task go:vet  # go vet ./...
task build   # go build -o ./dist/bx-pack ./cmd/bx-pack
task check   # go:vet + lint + test — быстрая проверка перед коммитом
task ci      # полный пайплайн: fmt + vet + lint + test
```

## Специфика проекта

### Bitrix-модули
- Валидация структуры модулей
- Упаковка с учётом исключений
- Конфигурация через `.bxpack.yml`
- `outputDir` не должен рекурсивно попадать в сборку

### Процесс упаковки
1. Полная валидация
2. Staging
3. Копирование
4. Архивация
5. Исключения трактуются единообразно

### CLI и UX
- Все пользовательские тексты — на русском
- Ошибки короткие и предметные
- Summary лаконичный и полезный
- JSON-вывод для машинного чтения
- Сохранять совместимость флагов и команд

## Тесты

При любом нетривиальном изменении думать о тестах.

### Требования
- Тестировать поведение и контракт, а не реализацию
- Для файловых сценариев использовать `t.TempDir()`
- Не использовать сеть, внешние сервисы, случайность
- Покрывать: happy path, основные ошибки, исключения
